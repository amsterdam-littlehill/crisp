# Workflow: Add Feature

## 前置检查（必须先读）

1. `rules/project-rules.md`
2. `rules/coding-standards.md`
3. `references/gotchas.md`（确认是否有相关坑点）

## 执行步骤

### 1. 规划

- 理解用户需求的边界——什么在范围内，什么不在
- 评估影响范围：改哪些文件、是否涉及接口变更
- 如果涉及 ≥3 个独立文件/模块 → 触发 `workflows/update-rules.md` 的多子任务流程

### 2. 实现

- 按 `rules/coding-standards.md` 的规范编写
- 遵循 `rules/project-rules.md` 的架构约定
- **精准修改**：不顺便重构无关代码、不添加未要求的功能

### 3. 测试

- <!-- FILL: 项目特定的测试命令 -->
- 确认新功能按预期工作
- 确认没有破坏现有功能

### 4. 30 秒 AAR 扫描

任务不算完成，直到回答这 4 个问题：

1. **新模式？** — 用了未记录的模式或约定吗？
2. **新陷阱？** — 遇到了不提前知道就会浪费大量时间的问题吗？
3. **缺失规则？** — 因为缺少某条规则导致走了弯路吗？
4. **过时规则？** — 发现现有规则已经不准确或不再适用吗？

**跳过条件**：纯格式化、纯注释、仅依赖版本变更、无新教训的重构。

如果任一答案为"是" → 走 `workflows/update-rules.md` 录入流程。
