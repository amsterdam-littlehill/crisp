# Workflow: Update Rules — Task Closure Protocol

> 任务主体完成后，必须执行的扫尾协议。不是可选的 polish，是任务定义的一部分。

## 完成标准

一个任务在以下条件**全部满足**前不算完成：

1. 主体工作完成并验证（代码跑通、测试通过、功能交付）
2. **30 秒 AAR 扫描**（4 个问题 —— 全部"否"则到此结束）
3. 如果任何一个"是" → 通过录入标准 → 通过则记录

**任何 workflow 不得在跳过第 2 步的情况下声明"完成"。**

---

## AAR 的 4 个问题（30 秒扫完）

| # | 问题 | 如果"是" → |
|---|------|------------|
| 1 | **新模式？** | 用了未记录的模式或约定吗？ |
| 2 | **新陷阱？** | 遇到了不提前知道就会浪费大量时间的问题吗？ |
| 3 | **缺失规则？** | 因为缺少某条规则导致走了弯路吗？ |
| 4 | **过时规则？** | 发现现有规则已经不准确或不再适用吗？ |

**跳过条件**（窄且明确）：仅格式化、仅注释、仅依赖版本变更、无新教训的重构。

---

## Recording Threshold（2/3 录入标准）

不是所有发现都值得记录。录入前通过阈值过滤：

| 维度 | 通过标准 |
|------|---------|
| **可重复？** | 这个坑会在未来重复出现吗？ |
| **代价高？** | 不提前知道会浪费 >30 分钟调试吗？ |
| **代码不可见？** | 从代码本身看不出来这个坑吗？ |

**至少 2/3 通过才录入。**

### 录入位置判断

| 内容类型 | 目标位置 |
|---------|---------|
| 稳定约束 / 通用原则 | `rules/` |
| 陷阱、架构笔记、生命周期坑 | `references/` + SKILL.md Known Gotchas |
| 有序步骤 / 完成检查清单 | `workflows/` |
| 会话历史 / 调试过程 | **不要写进 skill** — 用 git / CHANGELOG |

---

## Generalization Rule（泛化规则）

**记录的内容必须脱离当前项目上下文也能看懂。**

改写公式：
```
具体发现 → 抽象为通用 pattern → 说明不遵守的后果
```

---

## Rationalizations to Reject

压力下 Agent 会生成借口绕过协议。以下是常见借口——**拒绝它们**：

| 借口 | 反驳 |
|------|------|
| "这次改动太小了，不需要 AAR" | 跳过条件只有 4 种：纯格式化、纯注释、纯依赖版本、无新教训重构。你的任务不属于这 4 种。 |
| "这个坑很明显，不需要记录" | "很明显"是事后视角。下次踩到它的 Agent 不会觉得明显。 |
| "我先记下来，晚点再整理" | 不会整理。现在就用 2/3 标准判断，要么立刻录入，要么丢弃。 |
| "这条规则太具体了，没法泛化" | 如果无法泛化 → 说明它不满足 2/3 标准 → 不录入。 |

**硬约束**：这张表只能从真实失败里加行，禁止凭空扩写。

---

## Red Flags — STOP

以下任何一条出现，立刻停下，不要自我协商：

- 发现自己在想"这次 AAR 就算了"
- 任务声明"完成"但没跑 30 秒扫描
- 把 gotcha 写进了 reference，但没更新对应 workflow 的完成清单
- 修了同一类 bug 第二次，但规则文件没动过
